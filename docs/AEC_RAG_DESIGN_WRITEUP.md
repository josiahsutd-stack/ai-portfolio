# Designing A Source-Grounded AEC RAG Assistant

The flagship project is an AEC code-compliance RAG assistant. It does not replace an architect, engineer, code consultant, or authority reviewer. Its scope is narrower: retrieve relevant guidance, expose citations, abstain when evidence is missing, and keep each answer inspectable.

AEC information work is a good RAG problem because the risk is not only whether a model sounds fluent. The risk is whether it can show where an answer came from. A planning note, accessibility clause, fire-safety requirement, drainage guide, or drawing QA checklist can be versioned, jurisdiction-specific, superseded, or simply outside the local corpus. A generic chatbot pattern is weak here because a confident answer without source context is worse than no answer. The project therefore treats retrieval and evidence display as first-class behavior, not as a decorative add-on after generation.

The ingestion layer supports Markdown and text-based PDFs. Documents are converted into chunks with metadata: source ID, title, document type, jurisdiction, section, page marker, chunk ID, and source-status fields such as superseded or mixed-version warnings. This metadata matters because AEC review is rarely just semantic similarity. Jurisdiction filters, superseded-document warnings, and source-type labels distinguish synthetic demo guidance from downloaded public sources. The Singapore public-source downloader rejects HTML error pages presented as PDFs and fingerprints every accepted file. Downloaded BCA, URA, NEA, SCDF, LTA, PUB, and NParks files stay local and are not redistributed in the repository.

Retrieval is implemented as several inspectable baselines. TF-IDF gives a simple lexical baseline that is fast and deterministic. BM25 improves term-frequency handling and is useful for clause-like language where exact wording matters. Dense LSA adds a lightweight semantic retrieval path without requiring hosted embeddings. Hybrid retrieval combines lexical and dense signals so the system can use both exact terminology and broader phrasing. The public evaluation reports document discovery separately from manually labeled exact-chunk and source-page targets: document Hit@1 is `0.952`, exact-target Hit@1 is `0.810`, and page-target Hit@1 is `0.778`. This gap prevents a correct publication from being mistaken for a correct supporting passage. The labels are fixed-snapshot project annotations, not expert clause labels.

The answer layer is intentionally conservative. Every answer is built around retrieved evidence and citation objects. A citation includes a readable reference, source metadata, score, excerpt, and chunk ID. The assistant can also return a no-answer response when the corpus does not support the question. That abstention behavior matters. In many AEC workflows, the correct behavior is not to improvise, but to say the local sources do not contain enough evidence and name the missing scope. The project tests this no-answer path alongside normal answerable questions.

Evaluation is treated as a product feature. The harness includes synthetic AEC questions, expected sources, expected sections, answerability status, and citation checks. Metrics include recall at k, precision at k, hit rate, mean reciprocal rank, hit-at-1, hit-at-3, citation coverage, status accuracy, no-answer accuracy, and unsupported-scope accuracy. These metrics do not prove real-world compliance performance. They show whether the local system retrieves the intended evidence, cites it, and refuses unsupported questions consistently. Versioned demo outputs and ablation reports make the generated evidence directly inspectable.

The strongest engineering choice in this project is the boundary around claims. The assistant is a source-grounded document workflow, not legal or professional advice. The default corpus is synthetic so the repo can run without private data or hidden services. The public-source workflow is optional and local. Mock provider behavior keeps tests deterministic without paid API keys. These constraints make the project less flashy, but more reviewable.

The local service also makes its reliability boundary testable. Process counters are paired with bounded payload-free SQLite request rows, query objectives remain `insufficient_data` below a minimum sample, and a fixed concurrent workload reconstructs the app to verify persistence. The main limitations remain clear: there is no OCR for scanned PDFs, authority validation, distributed production monitoring, user study, network load evidence, or guarantee that public documents remain current. A deployable version would need stronger document governance, identity-aware authorization, distributed telemetry, reverse-proxy and security testing, operational source refresh, and expert validation. The project demonstrates the applied pattern through executable evidence: ingest documents, preserve metadata, retrieve and cite evidence, abstain when unsupported, and measure both retrieval and local service behavior.
