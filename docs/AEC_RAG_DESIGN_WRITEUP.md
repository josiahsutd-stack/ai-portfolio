# Designing A Source-Grounded AEC RAG Assistant

The flagship project in this portfolio is an AEC code-compliance RAG assistant. The goal is not to replace an architect, engineer, code consultant, or authority reviewer. The goal is narrower and more useful for an applied AI engineering portfolio: show how a local assistant can retrieve relevant guidance, expose citations, abstain when evidence is missing, and make every answer inspectable.

AEC information work is a good RAG problem because the risk is not only whether a model sounds fluent. The risk is whether it can show where an answer came from. A planning note, accessibility clause, fire-safety requirement, drainage guide, or drawing QA checklist can be versioned, jurisdiction-specific, superseded, or simply outside the local corpus. A generic chatbot pattern is weak here because a confident answer without source context is worse than no answer. The project therefore treats retrieval and evidence display as first-class behavior, not as a decorative add-on after generation.

The ingestion layer supports Markdown and PDF-like sources. Documents are converted into chunks with metadata: source ID, title, document type, jurisdiction, section, page marker, chunk ID, and source-status fields such as superseded or mixed-version warnings. This metadata is important because AEC review is rarely just semantic similarity. A reviewer may need to filter by jurisdiction, avoid superseded documents, or understand that evidence came from a synthetic demo source rather than a downloaded public source. The project also includes an optional Singapore public-source workflow for BCA, URA, NEA, SCDF, LTA, PUB, and NParks materials. The downloaded public files stay local and are not redistributed in the repository.

Retrieval is implemented as several inspectable baselines. TF-IDF gives a simple lexical baseline that is fast and deterministic. BM25 improves term-frequency handling and is useful for clause-like language where exact wording matters. Dense LSA adds a lightweight semantic retrieval path without requiring hosted embeddings. Hybrid retrieval combines lexical and dense signals so the system can benefit from both exact terminology and broader phrasing. The point is not to claim frontier retrieval quality. The point is to show retrieval engineering discipline: multiple modes, comparable metrics, and enough transparency that a reviewer can see why a chunk was selected.

The answer layer is intentionally conservative. Every answer is built around retrieved evidence and citation objects. A citation includes a readable reference, source metadata, score, excerpt, and chunk ID. The assistant can also return a no-answer response when the corpus does not support the question. That abstention behavior matters. In many AEC workflows, the correct behavior is not to improvise, but to say the local sources do not contain enough evidence and name the missing scope. The project tests this no-answer path alongside normal answerable questions.

Evaluation is treated as a product feature. The evaluation harness includes synthetic AEC questions, expected sources, expected sections, answerability status, and citation checks. Metrics include recall at k, precision at k, hit rate, mean reciprocal rank, hit-at-1, hit-at-3, citation coverage, status accuracy, no-answer accuracy, and unsupported-scope accuracy. These metrics do not prove real-world compliance performance. They do show whether the local system retrieves the intended evidence, cites it, and refuses unsupported questions consistently. The repository also writes demo outputs and ablation reports so reviewers can inspect generated evidence rather than trusting a README claim.

The strongest engineering choice in this project is the boundary around claims. The assistant is a source-grounded document workflow, not legal or professional advice. The default corpus is synthetic so the repo can run without private data or hidden services. The public-source workflow is optional and local. Mock provider behavior keeps tests deterministic without paid API keys. These constraints make the project less flashy, but more reviewable.

The main limitations are clear. There is no OCR for scanned PDFs, no authority validation, no production monitoring, no user study, and no guarantee that public documents are current after download. The retrieval stack is lightweight compared with a production search system. A deployable version would need stronger document governance, version tracking, OCR, structured clause parsing, semantic reranking, security review, usage logging, and expert validation. Still, as a portfolio project, the system demonstrates the core applied AI pattern well: ingest documents, preserve metadata, retrieve evidence, cite sources, abstain when unsupported, and evaluate the behavior repeatedly.
